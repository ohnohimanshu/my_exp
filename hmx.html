Experiment 1: Enterprise-Grade SSH Implementation

Secure Shell (SSH) is a cryptographic network protocol that provides a secure channel over an unsecured network. It allows for secure remote login, command execution, and file transfers between networked machines.

In enterprise environments, SSH is the backbone of secure infrastructure management, allowing administrators to securely access and manage production servers, cloud instances, and network devices.

ssh-connection.sh
$ ssh-keygen -t ed25519 -C "admin@enterprise.com"
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/user/.ssh/id_ed25519): 
Enter passphrase (empty for no passphrase): 
Enter same passphrase again: 
Your identification has been saved in /home/user/.ssh/id_ed25519
Your public key has been saved in /home/user/.ssh/id_ed25519.pub

Installing OpenSSH Server and Client

The OpenSSH implementation is the industry standard for secure remote access. Begin by installing both server and client components:

$ 
sudo apt update
sudo apt install openssh-server -y
sudo systemctl enable ssh
sudo systemctl start ssh
sudo systemctl status ssh

The commands above perform the following operations:

  • Update your system's package repository information
  • Install the OpenSSH server package
  • Configure the SSH service to start automatically at boot time
  • Start the SSH service immediately
  • Verify the service is running correctly

Network Identification and Connectivity

To establish a connection, you'll need to identify your system's IP address:

# Display all network interfaces and their IP addresses
ip a

# Alternative command to show only IP addresses
hostname -I

Establishing Secure Connections

Once the server is configured, you can connect from another machine using:

ssh username@ip-address

The first time you connect to a new server, you'll be prompted to verify the host's authenticity by accepting its fingerprint. This is a critical security step to prevent man-in-the-middle attacksAn attack where an attacker secretly relays and possibly alters communication between two parties who believe they are directly communicating with each other..

Implementing Enhanced Security Measures

For enterprise environments, implement these critical security enhancements:

Key-Based Authentication

Eliminate password vulnerabilities by implementing key-based authentication:

# Generate a secure 4096-bit RSA key
ssh-keygen -t rsa -b 4096 -C "username@company.com"

# Copy public key to the remote server
ssh-copy-id username@ip-address

# Test passwordless authentication
ssh username@ip-address

Hardening SSH Configuration

Edit the SSH server configuration to enhance security:

sudo nano /etc/ssh/sshd_config

Recommended security settings:

# Disable root login
PermitRootLogin no

# Use a non-standard port
Port 2222

# Disable password authentication
PasswordAuthentication no

# Limit authentication attempts
MaxAuthTries 3

# Restrict SSH protocol version
Protocol 2

# Set idle timeout (in seconds)
ClientAliveInterval 300
ClientAliveCountMax 2

After making changes, restart the SSH service:

sudo systemctl restart ssh
When changing the SSH port or disabling password authentication, ensure you have an alternate way to access the system in case of configuration errors. Always keep a session open when testing SSH configuration changes.

Experiment 2: Data Analysis Environment with Jupyter Notebook

Jupyter Notebook provides an interactive computing environment that enables data scientists, analysts, and researchers to create and share documents containing live code, equations, visualizations, and narrative text.

In enterprise settings, Jupyter serves as a critical tool for data exploration, model development, visualization, and collaborative analysis. This experiment outlines a production-ready Jupyter implementation with remote access capabilities and security enhancements.

Installing Python and Dependencies

Begin by setting up the Python environment that will host Jupyter:

$ 
sudo apt update

sudo apt install python3-dev -y

Installing Jupyter and Essential Data Science Libraries

Install Jupyter and common data science packages:

pip3 install notebook jupyterlab

# Install essential data science packages
pip3 install numpy pandas matplotlib scikit-learn seaborn plotly

Configuring Remote Access

To enable secure remote access, configure Jupyter to accept connections from external IP addresses:

# Generate configuration file
jupyter notebook --generate-config

# Set a secure password
jupyter notebook password

You'll be prompted to create and confirm a password. This password will be required when accessing your Jupyter server remotely.

Customizing Jupyter Configuration

Modify the Jupyter configuration file to enable remote access with enhanced security:

nano ~/.jupyter/jupyter_notebook_config.py

Add or modify these settings:

# Allow all IP addresses
c.NotebookApp.ip = '0.0.0.0'

# Don't open browser automatically
c.NotebookApp.open_browser = False

# Set the port
c.NotebookApp.port = 8888

# Use HTTPS
c.NotebookApp.certfile = '/home/username/ssl/mycert.pem'
c.NotebookApp.keyfile = '/home/username/ssl/mykey.key'

# Configure authentication
c.NotebookApp.password_required = True

# Set notebook autosave interval in seconds
c.FileContentsManager.autosave_interval = 120

Creating SSL Certificates for HTTPS

For secure HTTPS connections, generate self-signed SSL certificates:

mkdir -p ~/ssl
cd ~/ssl
openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout mykey.key -out mycert.pem

Running Jupyter as a Background Service

For production environments, create a systemd service to run Jupyter in the background:

sudo nano /etc/systemd/system/jupyter.service

Add the following configuration:

[Unit]
Description=Jupyter Notebook Server
After=network.target

[Service]
Type=simple
User=username
ExecStart=/usr/local/bin/jupyter notebook --config=/home/username/.jupyter/jupyter_notebook_config.py
WorkingDirectory=/home/username/notebooks
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
$

Configuring Firewall Access

Allow traffic to the Jupyter port:

sudo ufw allow 8888

Accessing Jupyter Remotely

Access your Jupyter server from another machine by navigating to:

https://server-ip-address:8888
For production environments with sensitive data, consider additional security measures such as implementing a VPN, restricting IP ranges, or using an authentication proxy. Regularly update libraries to mitigate security vulnerabilities.

Experiment 3: Enterprise Web Application Deployment with Apache Tomcat

Apache Tomcat is a robust, production-grade web server and servlet container that provides a "pure Java" HTTP web server environment for running Java applications. It implements several Java EE specifications including Java Servlet, JavaServer Pages (JSP), and WebSocket.

In enterprise environments, Tomcat is commonly used to host business-critical applications, REST APIs, and microservices. This experiment outlines a comprehensive setup with security hardening and performance optimization suitable for production use.

Tomcat 10.1.36 Java 11 Servlet 5.0 JSP 3.0

Installing Java Development Kit

Tomcat requires a Java Development Kit (JDK) to function. Install OpenJDK 11:

sudo apt update
sudo apt install openjdk-11-jdk -y

# Verify installation
java --version

Creating a Dedicated Tomcat User

For enhanced security, create a dedicated non-privileged user for running Tomcat:

sudo groupadd tomcat
sudo useradd -s /bin/false -g tomcat -d /opt/tomcat tomcat

Downloading and Installing Apache Tomcat

Download and extract the latest stable version of Tomcat:

$ 
cd /tmp
wget https://dlcdn.apache.org/tomcat/tomcat-10/v10.1.36/bin/apache-tomcat-10.1.36.tar.gz
sudo mkdir -p /opt/tomcat
sudo tar -xzvf apache-tomcat-10.1.36.tar.gz -C /opt/tomcat --strip-components=1

Setting Appropriate Permissions

Configure directory permissions for secure operation:

cd /opt/tomcat
sudo chgrp -R tomcat /opt/tomcat
sudo chmod -R g+r conf
sudo chmod g+x conf
sudo chown -R tomcat webapps/ work/ temp/ logs/
sudo chmod +x /opt/tomcat/bin/*.sh

Creating a Systemd Service for Tomcat

Create a systemd service file to manage Tomcat as a service:

sudo nano /etc/systemd/system/tomcat.service

Add the following configuration:

[Unit]
Description=Apache Tomcat Web Application Container
After=network.target

[Service]
Type=forking

Environment="JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64"
Environment="CATALINA_PID=/opt/tomcat/temp/tomcat.pid"
Environment="CATALINA_HOME=/opt/tomcat"
Environment="CATALINA_BASE=/opt/tomcat"
Environment="CATALINA_OPTS=-Xms512M -Xmx1024M -server -XX:+UseParallelGC"
Environment="JAVA_OPTS=-Djava.awt.headless=true -Djava.security.egd=file:/dev/./urandom"

ExecStart=/opt/tomcat/bin/startup.sh
ExecStop=/opt/tomcat/bin/shutdown.sh

User=tomcat
Group=tomcat
UMask=0007
RestartSec=10
Restart=always

[Install]
WantedBy=multi-user.target

Starting and Enabling Tomcat Service

Start Tomcat and enable it to launch at boot:

sudo systemctl daemon-reload
sudo systemctl start tomcat
sudo systemctl enable tomcat
sudo systemctl status tomcat

Deploying a Sample Web Application

Experiment 4: Local AI Model Deployment with Ollama

Overview

Ollama is a powerful framework that allows you to run large language models (LLMs) locally. This capability enables organizations to maintain control over their data while leveraging advanced AI capabilities for content generation, knowledge extraction, and custom applications.

This experiment demonstrates how to deploy, customize, and integrate open-source large language models into your enterprise environment, providing a secure alternative to cloud-based AI services.

ollama-chat.log
> How can I securely implement SSH in an enterprise environment?

SSH (Secure Shell) implementation in enterprise environments requires several security considerations:

1. Key-based authentication: Disable password authentication and use SSH keys with passphrases.
2. Non-standard ports: Change from default port 22 to reduce automated attacks.
3. Restricted user access: Implement least privilege principle with specific users.
4. Two-factor authentication: Add an extra layer of security beyond keys.
5. Regular updates: Keep SSH software and configurations updated.
6. Firewall rules: Restrict SSH access to specific IP ranges.
7. Jump servers: Implement centralized access points for better auditing.
8. Session timeouts: Configure automatic disconnection for idle sessions.
9. Failed login limits: Implement account lockouts after multiple failures.
10. Logging and monitoring: Track all authentication attempts.

For implementation, I recommend updating /etc/ssh/sshd_config with:
```
PasswordAuthentication no
PermitRootLogin no
Protocol 2
MaxAuthTries 5
ClientAliveInterval 300
ClientAliveCountMax 0
```

Installing Prerequisites

Install essential packages for Ollama deployment:

sudo apt update
sudo apt install git curl build-essential -y

Installing Ollama

Download and install Ollama using the official installation script:

$

Verifying the Installation

Verify that Ollama is running correctly:

ollama --version

Running Pre-trained Language Models

Ollama provides access to various pre-trained models. Download and run a lightweight model like Llama3:

# Download and run the 2.1B parameter version of Llama3
ollama run llama3:2.1b

The first time you run this command, Ollama will download the model. Once downloaded, you'll enter an interactive chat session where you can communicate with the AI model.

Managing Available Models

List and manage your installed models:

# List available models
ollama list

# Pull a model without running it
ollama pull tinyllama

# Remove a model
ollama rm llama3:2.1b

Creating Custom AI Models

Create specialized models tailored to your enterprise needs using a Modelfile:

mkdir -p ~/ollama-models
cd ~/ollama-models
nano Modelfile

Define your custom model specifications:

FROM llama3:2.1
PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER top_k 40
SYSTEM """
You are a helpful enterprise assistant specialized in technology documentation.
Always provide accurate, concise information with practical examples.
Prioritize security best practices in your responses.
Format code examples clearly with appropriate syntax highlighting.
"""

Build your custom model:

ollama create tech-assistant -f ./Modelfile

Setting Up the Ollama API Server

For integration with applications, run Ollama as an API server:

ollama serve

The API server runs on port 11434 by default. You can interact with it using HTTP requests:

curl -X POST http://localhost:11434/api/generate -d '{
  "model": "tech-assistant",
  "prompt": "What are the best practices for SSH security?",
  "stream": false
}'

Creating a Systemd Service for Ollama

For production use, create a systemd service to run Ollama in the background:

sudo nano /etc/systemd/system/ollama.service

Configure the service:

[Unit]
Description=Ollama AI Model Server
After=network.target

[Service]
User=username
ExecStart=/usr/local/bin/ollama serve
Restart=always
RestartSec=10
Environment="OLLAMA_MODELS=/home/username/ollama-models"

[Install]
WantedBy=multi-user.target
For production environments, consider implementing additional security measures such as running Ollama behind a reverse proxy with TLS/SSL, implementing authentication, and restricting access to authorized IP addresses only.

Experiment 5: Distributed Data Processing with Hadoop

Overview

Apache Hadoop is an open-source framework that allows for distributed processing of large data sets across clusters of computers. It is designed to scale up from single servers to thousands of machines, offering local computation and storage.

This experiment demonstrates how to set up a Hadoop environment suitable for enterprise big data processing, including HDFS (Hadoop Distributed File System) for storage and YARN (Yet Another Resource Negotiator) for resource management.

Creating a Dedicated Hadoop User

Create a dedicated user for Hadoop operations to enhance security and separation of concerns:

Configuring SSH for Hadoop User

Set up passwordless SSH for the Hadoop user, which is required for Hadoop services to communicate:

# Switch to hadoop user
sudo su - hadoop

# Generate SSH keys without passphrase
ssh-keygen -t rsa -P '' -f ~/.ssh/id_rsa

Configuring Core Hadoop Files

Configure essential Hadoop configuration files:

1. Setting Environment Variables

sudo su - hadoop
nano ~/.bashrc

Add these environment variables to the end of the file:

# Hadoop Environment Variables
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
export HADOOP_HOME=/home/hadoop/hadoop
export HADOOP_INSTALL=$HADOOP_HOME
export HADOOP_MAPRED_HOME=$HADOOP_HOME
export HADOOP_COMMON_HOME=$HADOOP_HOME
export HADOOP_HDFS_HOME=$HADOOP_HOME
export YARN_HOME=$HADOOP_HOME
export HADOOP_COMMON_LIB_NATIVE_DIR=$HADOOP_HOME/lib/native
export PATH=$PATH:$HADOOP_HOME/sbin:$HADOOP_HOME/bin
export HADOOP_OPTS="-Djava.library.path=$HADOOP_HOME/lib/native"
# Apply the changes
source ~/.bashrc

2. Configuring hadoop-env.sh

nano $HADOOP_HOME/etc/hadoop/hadoop-env.sh

Set the JAVA_HOME variable by adding or updating:

export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64

3. Configuring core-site.xml

nano $HADOOP_HOME/etc/hadoop/core-site.xml

Add these configuration properties:

<configuration>
    <property>
        <name>fs.defaultFS</name>
        <value>hdfs://localhost:9000</value>
    </property>
    <property>
        <name>hadoop.tmp.dir</name>
        <value>/home/hadoop/hadoop_tmp</value>
    </property>
</configuration>

4. Configuring hdfs-site.xml

nano $HADOOP_HOME/etc/hadoop/hdfs-site.xml

Add these configuration properties:

<configuration>
    <property>
        <name>dfs.replication</name>
        <value>1</value>
    </property>
    <property>
        <name>dfs.namenode.name.dir</name>
        <value>/home/hadoop/hadoop_data/hdfs/namenode</value>
    </property>
    <property>
        <name>dfs.datanode.data.dir</name>
        <value>/home/hadoop/hadoop_data/hdfs/datanode</value>
    </property>
</configuration>

5. Configuring mapred-site.xml

nano $HADOOP_HOME/etc/hadoop/mapred-site.xml

Add these configuration properties:

<configuration>
    <property>
        <name>mapreduce.framework.name</name>
        <value>yarn</value>
    </property>
    <property>
        <name>mapreduce.application.classpath</name>
        <value>$HADOOP_HOME/share/hadoop/mapreduce/*:$HADOOP_HOME/share/hadoop/mapreduce/lib/*</value>
    </property>
</configuration>

6. Configuring yarn-site.xml

nano $HADOOP_HOME/etc/hadoop/yarn-site.xml

Add these configuration properties:

<configuration>
    <property>
        <name>yarn.nodemanager.aux-services</name>
        <value>mapreduce_shuffle</value>

</configuration>
